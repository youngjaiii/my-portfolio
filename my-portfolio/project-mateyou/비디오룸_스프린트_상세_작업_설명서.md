# 비디오룸 전환 스프린트 상세 작업 설명서

본 문서는 비디오룸 구조 전환 로드맵의 Sprint 작업을 실행 가능한 수준으로 상세화한 것입니다.  
각 항목은 “무엇을 해야 하는지” 기준으로 정리했으며, 인프라/백엔드/프론트 관점의 작업을 함께 포함합니다.

---

## 1. 전체 시스템 아키텍처

### 1.1 논리 구조

```
[송출자]
  ├─ WebRTC (웹 / 모바일)
  └─ RTMP (OBS / PRISM)
        ↓
[LiveKit] / [Nginx RTMP]
        ↓
[FFmpeg]
        ↓
[HLS]
        ↓
[S3 + CloudFront]
        ↓
[시청자]
```

### 1.2 구성 요소 개요

| 구성 요소 | 역할 |
|---|---|
| Frontend (React) | 송출 UI, 시청 UI |
| LiveKit | WebRTC 송출 수신 |
| Backend API | 방송 제어, 상태 관리 |
| Nginx RTMP | RTMP 수신 |
| FFmpeg | HLS 변환 |
| S3 | HLS 저장 |
| CloudFront | 시청 트래픽 처리 |
| Supabase | 인증, 채팅 |

---

## 2. 스프린트 순서(요구사항/종속성 기준)

요구사항과 종속성을 고려하면 개발 순서는 아래가 자연스럽습니다.

1) **서버/인프라를 먼저 “싹” 구축**: LiveKit/FFmpeg/Nginx RTMP/S3/CloudFront(보안/관측 포함)  
2) **송출부 코어 개발**: 토큰/제어 API + WebRTC(LiveKit) 송출 기능  
3) **송출부 종류별 확장**: RTMP(OBS/PRISM) 경로 추가(스트림키/인증 포함)  
4) **수신부 개발**: HLS URL/API + 플레이어 + UX/안정성  
5) **점진 전환/정리**: Feature Flag, PeerJS 경로 정리, 통합 테스트

---

## 3. 로드맵 (Sprint 1-8)

## Phase 0: 서버/인프라 구축 (Sprint 1)

### Sprint 1: 인프라 기반 “일괄” 구축

#### 1) LiveKit 서버 인프라 구성(배포, 보안, 모니터링 기본)
- 인프라 아키텍처 결정(단일/다중 노드, VM/컨테이너, 로드밸런서/도메인)
- 배포 방식 확정(예: Docker Compose/Kubernetes) 및 재시작/롤링 전략 정의
- 도메인/SSL 구성 및 방화벽/보안 그룹 설정
- LiveKit 서버 설정 정리(포트, API Key/Secret, 로깅, 레이트리밋/보안 옵션)
- 관측 기본 구성(헬스체크, CPU/메모리, 네트워크, 프로세스 다운 알람)
- 장애 대응 절차 문서화(재시작, 설정 롤백, 키 회전)

#### 2) FFmpeg 서버 인프라 구성(배포, 스토리지 경로 설계)
- 실행 환경 구성(전용 서버/컨테이너, 리소스 제한, 권한)
- HLS 출력 저장 방식 확정(로컬 디스크/볼륨 마운트/S3 업로드)
- 저장 경로/명명 규칙 정의(방 ID 기준 디렉토리, m3u8/segment 파일 규칙)
- HLS 생성 정책 기본값 정리(세그먼트 길이, playlist 길이, 삭제 정책)
- 프로세스/로그 수집 기본 구성(실행 로그, 실패 원인 추적)

#### 3) Nginx RTMP 서버 인프라 구성(배포, 보안, 모니터링 기본)
- RTMP 인바운드 기본 설정(rtmp listen, application) 및 접근 제어
- on_publish 인증 훅 연동 방식 확정(HTTP callback, 타임아웃/재시도 정책)
- 기본 로그/메트릭 수집 구성(접속/거부/끊김, 대역폭)

#### 4) S3 + CloudFront 기본 구성(수신 트래픽 경로 선행 확보)
- S3 버킷 생성(HLS 저장), CORS/접근 정책 확정(권장: CloudFront OAC/OAI)
- CloudFront 배포 생성(S3 origin), 캐시/TTL 기본값 및 HLS 스트리밍 옵션 정리
- HLS 경로 규칙과 CloudFront URL 형태 확정(방 ID 기반)

## Phase 1: 송출부 코어 개발 (Sprint 2-3)

### Sprint 2: Backend 코어(API 뼈대 + LiveKit 토큰/권한)

#### 1) Backend API 설계 및 엔드포인트 스켈레톤
- 토큰 발급 입력/출력 정의(방 ID, 사용자 ID, 역할/권한)
- 인증 방식 결정(Supabase 인증 연동 방식 포함)
- API 호스팅/라우팅 결정(Node.js API vs Edge Functions 확장)
- 엔드포인트 스켈레톤 구현(라우팅/컨트롤러/서비스 레이어)
- 환경 변수/시크릿 설계(LiveKit API Key/Secret, RTMP 검증용 시크릿 등)
- 에러 응답 포맷 표준화(권한 오류, 방 없음 등)

#### 2) LiveKit 토큰 발급 API 구현(방 ID, 권한 포함)
- 토큰 생성 로직 구현(LiveKit Server SDK 활용)
- 호스트/시청자 권한 모델 확정(publish/subscribe 등)
- 방 존재/상태 검증 로직 추가(기존 방 관리 API와 연동)
- 응답 페이로드 확정(token, room, expiresAt 등)
- 기본 테스트(정상/권한 없음/방 없음)

#### 3) 방송 제어 API(최소 코어)
- 방송 시작/종료(호스트 권한) 엔드포인트 정의
- 방송 상태 조회(시청자/관리자) 엔드포인트 정의
- FFmpeg 제어는 “연동 지점(인터페이스)”만 먼저 설계(실행/운영은 Sprint 4에서)

#### 4) 방송 상태 머신(FSM) 최소 구현
- 상태/전이 정의(scheduled → live → ended, 오류 상태 포함 여부)
- 상태 변경 트리거 정의(송출 시작/종료, 강제 종료, 오류)
- 상태 변경 로그/감사 이벤트 기록(최소)

### Sprint 3: WebRTC 송출 코어(Frontend Host)

#### 1) `VideoRoomProvider.tsx` 송출 로직 LiveKit 연동
- 기존 PeerJS 송출 로직 분석 및 LiveKit SDK 매핑
- LiveKit Room 생성/조인 기본 흐름 구현
- 로컬 미디어 트랙 생성 및 publish 처리
- 송출 UI/상태 연결(연결 중, 연결 완료, 실패)

#### 2) LiveKit Room 연결/해제/재연결 코어
- Room connect/disconnect 핸들링
- 연결 실패/타임아웃 처리 기본 로직
- 정상 종료 시 리소스 정리(트랙 unpublish, media stream 해제)
- 재연결 정책 정의(백오프/최대 횟수) 및 UI 피드백

#### 3) 화면 공유 포함(송출 코어 완성)
- 화면 공유 트랙 생성 및 publish
- 카메라/화면 전환 UX 처리(상태 관리, 토글)
- 기본 오디오 트랙 처리 및 음소거 상태 연동

## Phase 2: WebRTC → HLS 파이프라인(코어) (Sprint 4)

### Sprint 4: LiveKit → FFmpeg → HLS 통합 및 저장 안정화

#### 1) LiveKit → FFmpeg 입력 전달 방식 확정/구현
- LiveKit에서 FFmpeg로 입력 전달 방식 결정 및 구현(수신 방식/권한/네트워크)
- 방송 시작/종료 API와 FFmpeg 실행/종료 연동(초기 버전)
- 장애 시 처리(FFmpeg 다운/입력 끊김) 정책 정의

#### 2) HLS 세그먼트/매니페스트 생성 안정화
- 세그먼트 누락/지연 발생 시 처리 정책
- m3u8 갱신 주기 및 플레이리스트 유지 길이 조정
- 저장 공간 정리 정책(지난 세그먼트 삭제 규칙)

#### 3) S3 업로드 + CloudFront 경로로 재생 확인(개발용)
- HLS 산출물을 S3에 저장/업로드하는 방식 확정(직접 업로드/동기화)
- CloudFront URL로 최소 재생 확인(개발용 도구: `ffplay`/VLC 등 가능)

---

## Phase 3: 송출부 종류 확장(RTMP) (Sprint 5)

### Sprint 5: RTMP(OBS/PRISM) 경로 추가

#### 1) 스트림 키 발급/검증 API 구현
- 방 ID ↔ 스트림 키 매핑 로직 구현
- 키 발급/회수 정책 정의(유효기간, 재발급, 유출 대응)
- 키 검증 API 성능/보안 고려(레이트리밋, IP 제한 정책)

#### 2) Nginx RTMP on_publish 인증 훅 연동
- on_publish에서 스트림 키 검증 API 호출
- 실패/타임아웃 시 거부 동작 확정
- RTMP 로그/메트릭 수집 기준 확정

#### 3) RTMP → FFmpeg → HLS 파이프라인 구성
- RTMP 인입 스트림을 FFmpeg로 전달(입력 URL 규칙 포함)
- HLS 결과물을 기존 저장 규칙(S3/CloudFront)과 동일하게 맞춤
- OBS/PRISM 송출 시나리오로 E2E 검증

---

## Phase 4: 수신부(시청) 개발 (Sprint 6-7)

### Sprint 6: HLS 조회/API + 시청 UI 기본

#### 1) HLS URL 조회 API 구현(방 ID 기반)
- 방 ID → HLS 경로 매핑 로직 구현(CloudFront URL 기준)
- 상태 기반 응답(방 미생성/종료/라이브 중 여부)
- 라이브 중 m3u8 준비 전 상태 처리(준비중 응답/재시도 힌트)

#### 2) HLS 플레이어 기술 선택(HLS.js vs Video.js) 및 통합
- 지원 요구사항 정리(브라우저 범위, 커스터마이징, 번들 크기)
- 선택 근거 기록 후 컴포넌트/훅 형태로 통합(`HLSPlayer` 등)
- 자동재생 정책(브라우저 제한) 및 기본 오류 처리

#### 3) `$roomId.tsx` 시청 UI를 HLS로 전환(기본)
- HLS URL 조회 후 `<video>`에 연결
- 로딩/버퍼링/오류 UI 기본 구현

---

### Sprint 7: 시청 UX/안정성(재연결/버퍼링) + 관측

#### 1) 시청자 HLS 재연결/오류 정책 고도화
- 네트워크 오류 감지 시 자동 재시도(횟수/주기/백오프)
- 버퍼링 표시 및 장기 실패 시 사용자 안내
- 라이브 지연(레이턴시) 목표치에 맞춰 플레이어 옵션 튜닝

#### 2) 운영 로그/지표 최소 세트 추가(송출/시청 상태)
- 송출 상태 로그(연결, 재연결, 종료, 실패 원인)
- 시청 상태 로그(HLS 요청, 재생 오류, 재시도 횟수)
- 최소 대시보드/알람 구성(에러율, 동시 시청, FFmpeg 실패)

## Phase 5: 점진 전환/정리 (Sprint 8)

### Sprint 8: Feature Flag 전환 + 제거/안정화

#### 1) 기능 플래그로 병행 운용
- Feature Flag 설계 및 적용 범위 정의(내부/일부 사용자/전체)

- 제거 대상 파일/함수/의존성 목록화(시청 WebRTC 경로/PeerJS 서버 의존성)
