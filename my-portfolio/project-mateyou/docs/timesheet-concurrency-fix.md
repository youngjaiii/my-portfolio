# 근태 시스템 동시성 문제 해결

## 문제 상황

관리자가 강제로 상태를 변경하고 파트너+가 요청을 거의 동시에 보내면, 프론트엔드 상태가 변경되기 전에 요청이 전송되어 잘못된 상태 전이 요청이 생성되는 문제가 발생했습니다.

### 문제 시나리오

1. **관리자가 강제로 상태 변경**
   - `modifyAttendanceRecord` 또는 `createAttendanceRecord`로 파트너+ 상태를 WORKING → OFF로 변경

2. **파트너+의 프론트엔드 상태 불일치**
   - 프론트엔드는 아직 WORKING 상태를 표시
   - Realtime 업데이트가 아직 도착하지 않음

3. **잘못된 요청 전송**
   - 파트너+가 퇴근 요청(OFF)을 보냄
   - 서버에서는 이미 OFF 상태인데 퇴근 요청이 들어옴

4. **결과**
   - 잘못된 상태 전이 요청이 생성됨
   - 승인/거절 시 예상치 못한 동작 발생

## 해결 방안

### 1. 서버 측 상태 검증 추가 (`createAttendanceRequest`)

요청 생성 전에 서버의 현재 상태를 확인하고, 상태 전이 규칙에 맞는지 검증합니다.

**상태 전이 규칙:**
- `WORKING` 요청: 현재 상태가 `OFF`여야 함
- `BREAK` 요청: 현재 상태가 `WORKING`이어야 함
- `BREAK_END` 요청: 현재 상태가 `BREAK`이어야 함
- `OFF` 요청: 현재 상태가 `WORKING` 또는 `BREAK`이어야 함

**구현 내용:**
```typescript
// 1. 서버의 현재 상태 확인
const currentStatus = await getCurrentAttendanceStatus(partnerPlusId)

// 2. 상태 전이 규칙 검증
// 3. 대기 중인 요청 확인 (중복 요청 방지)
// 4. 검증 통과 시에만 요청 생성
```

### 2. 프론트엔드에서 최신 상태 확인

요청 전에 서버의 최신 상태를 확인하여 프론트엔드와 서버 상태 불일치를 감지합니다.

**구현 내용:**
```typescript
// 요청 전에 최신 상태 확인
const latestStatus = await getCurrentAttendanceStatus(user.id)

// 상태 불일치 시 경고 로그 (서버에서 다시 검증하므로 안전)
if (latestStatus !== currentStatus) {
  console.warn('⚠️ 상태 불일치 감지:', { frontend: currentStatus, server: latestStatus })
}
```

### 3. 중복 요청 방지

대기 중인 요청이 있는 경우 추가 요청을 차단합니다.

## 수정된 파일

1. **`src/lib/timesheetApi.ts`**
   - `createAttendanceRequest` 함수에 서버 상태 검증 로직 추가
   - 상태 전이 규칙 검증
   - 중복 요청 방지

2. **`src/components/features/timesheet/AttendanceRequestSheet.tsx`**
   - 요청 전 최신 상태 확인 로직 추가
   - 상태 불일치 감지 및 로깅

## 보호 메커니즘

### 다층 방어 체계

1. **프론트엔드 레벨**
   - 요청 전 최신 상태 확인
   - 상태 불일치 감지 및 로깅

2. **API 레벨**
   - 서버 상태 확인
   - 상태 전이 규칙 검증

3. **RPC 레벨** (기존)

## 에러 메시지

사용자에게 명확한 에러 메시지를 제공합니다:

- "출근 요청은 미출근 상태에서만 가능합니다. 현재 상태: 근무 중"
- "휴게 요청은 근무 중일 때만 가능합니다. 현재 상태: 이미 휴게 중"
- "휴게 해제 요청은 휴게 중일 때만 가능합니다. 현재 상태: 근무 중"
- "퇴근 요청은 근무 중이거나 휴게 중일 때만 가능합니다. 현재 상태: 이미 퇴근함"
- "이미 승인 대기 중인 요청이 있습니다. 요청이 처리될 때까지 기다려주세요."

## 테스트 시나리오

다음 시나리오를 테스트해야 합니다:

1. **관리자가 상태 변경 후 파트너+가 요청 전송**
   - 관리자가 WORKING → OFF로 변경
   - 파트너+가 퇴근 요청 시도
   - 에러 메시지 표시 및 요청 차단

2. **동시 요청 시도**
   - 파트너+가 연속으로 요청 전송
   - 첫 번째 요청이 대기 중일 때 두 번째 요청 차단

3. **정상적인 상태 전이**
   - OFF → WORKING → BREAK → BREAK_END → OFF
   - 각 단계에서 정상적으로 요청 생성 및 승인

## 향후 개선 사항

1. **RPC 함수 레벨 검증**
   - 요청 생성 시점에 데이터베이스 트리거로 검증 추가
   - 추가적인 안전장치 제공

2. **Realtime 업데이트 개선**
   - 상태 변경 시 즉시 프론트엔드 업데이트
   - WebSocket 또는 Supabase Realtime 활용

3. **낙관적 업데이트 제거**
   - 프론트엔드에서 낙관적 업데이트 사용 시 주의
   - 서버 상태를 항상 신뢰
