## 🎯 사용자 흐름

### 1. 호스트(방송하는 사람) 흐름

```
[방송 만들기] → 제목 정하고 공개/비공개 선택
      │
      ▼
[방송 시작] → 마이크 연결 → 인터넷 연결 → 30초마다 "나 살아있어요" 신호 보내기
      │
      ▼
[방송 중] → 청취자 관리, 채팅 관리, 후원 받기
      │
      ▼
[방송 끝] → 모든 연결 끊기 → 정리
```

### 2. 청취자(듣는 사람) 흐름

```
[방송 목록 보기] → 진행 중인 방송 찾기
      │
      ▼
[방송 입장] → 비밀번호 입력(필요시) → 인터넷 연결
      │
      ▼
[듣기] → 호스트 목소리 듣기 → 채팅하기 → 후원하기
      │
      ▼
[나가기] → "나 나갈게요" 알림 → 연결 끊기
```

### 🔴 문제 1: 새로고침하면 연결이 안 돼요 (PeerJS ID 충돌)

#### 😰 무슨 문제인가요?

인터넷 전화처럼 연결할 때 각 사람마다 **전화번호** 같은 고유 ID가 필요해요.

**문제가 생기는 상황:**
```
1. 철수가 방송에 들어가요 (ID: room1-철수)
2. 철수가 새로고침을 눌러요 (페이지가 다시 로드됨)
3. 그런데! 서버에는 아직 "room1-철수"가 연결되어 있다고 기록되어 있어요
4. 새로 접속한 철수가 같은 ID로 연결하려고 해요
5. 서버: "어? 이미 room1-철수가 있는데?" → 연결 거부! ❌
```

마치 **같은 전화번호로 두 개의 전화기를 동시에 쓰려는 것**과 같아요!

#### 😊 어떻게 해결하나요?

**현재 방법:**
```typescript
// 매번 새로운 번호를 붙여서 다른 ID를 만들어요
// 예: room1-철수-abc123 → room1-철수-xyz789
sessionIdRef.current = `${Date.now()}-${랜덤숫자}`
```

**더 좋은 방법 (연결 창고 사용):**
```typescript
// 연결을 창고에 보관해두고, 필요하면 재사용해요
class 연결창고 {
  창고 = new Map()
  
  연결가져오기(ID) {
    // 창고에 멀쩡한 연결이 있으면 그거 쓰기
    if (창고에_있고_아직_작동하면) {
      return 기존연결
    }
    // 없으면 새로 만들기
    return 새연결만들기()
  }
}
```

#### ✅ 해결 후 어떻게 달라지나요?

| 구분 | 해결 전 😰 | 해결 후 😊 |
|------|-----------|-----------|
| 새로고침 시 | "연결 실패" 에러 발생 | 부드럽게 다시 연결됨 |
| 사용자 경험 | 다시 방에 들어가야 함 | 자동으로 복구됨 |
| 에러 빈도 | 새로고침 시 자주 발생 | 거의 발생 안 함 |

**비유로 설명하면:**
- **해결 전**: 전화를 끊었다 다시 걸면 "이 번호는 사용 중입니다" 에러
- **해결 후**: 전화를 끊었다 다시 걸면 자연스럽게 연결됨

### 🟡 문제 2: 사람이 나갔는데 목록에 남아있어요 (참가자 목록 불일치)

#### 😰 무슨 문제인가요?

사람이 방송에서 나가면 다른 사람들에게 "나 나갔어!"라고 알려줘야 해요.

**문제가 생기는 상황:**
```
1. 영희가 방송을 듣고 있어요
2. 갑자기 영희 핸드폰 배터리가 꺼져요! 📱💀
3. "나 나갔어!" 메시지를 보낼 시간도 없이 꺼져버렸어요
4. 다른 사람들은 여전히 "영희가 참여 중"이라고 보여요
5. 영희한테 말을 걸어도 대답이 없어요 👻
```

마치 **교실에서 친구가 몰래 나갔는데, 출석부에는 아직 있다고 적혀 있는 것**과 같아요!

#### 😊 어떻게 해결하나요?

**해결책: 주기적으로 "나 살아있어요" 신호 보내기**
```typescript
// 30초마다 서버에 "나 아직 여기 있어요!" 신호를 보내요
매30초마다(() => {
  서버에_신호보내기("나 살아있어요!", 내ID)
})

// 서버에서는 2분 동안 신호가 없으면 자동으로 퇴장 처리해요
서버_청소작업 = () => {
  // 2분 동안 연락 없는 사람 찾기
  2분이상_연락없는_사람들 = 찾기()
  
  // 그 사람들 퇴장 처리
  2분이상_연락없는_사람들.forEach(사람 => {
    퇴장처리(사람)
  })
}
```

#### ✅ 해결 후 어떻게 달라지나요?

| 구분 | 해결 전 😰 | 해결 후 😊 |
|------|-----------|-----------|
| 배터리 꺼짐 시 | 계속 목록에 남아있음 | 2분 후 자동으로 사라짐 |
| 인터넷 끊김 시 | 유령처럼 목록에 남음 | 2분 후 자동으로 정리됨 |
| 참가자 수 표시 | 실제보다 많게 표시됨 | 정확하게 표시됨 |

**비유로 설명하면:**
- **해결 전**: 교실에서 친구가 나가도 선생님이 모름 (유령 출석)
- **해결 후**: 선생님이 1분마다 "철수야 여기있니?" 물어봐서 없는 애 바로 체크

### 🔴 문제 3: 오래 방송 들으면 앱이 느려져요 (메모리 누수)

#### 😰 무슨 문제인가요?

소리를 듣기 위해 **AudioContext**라는 도구를 사용해요. 그런데 이걸 잘못 쓰면 문제가 생겨요.

**문제가 생기는 상황:**
```
1. 첫 번째 호스트가 말해요 → AudioContext 1개 생성
2. 두 번째 호스트가 말해요 → AudioContext 또 1개 생성
3. 세 번째 호스트가 말해요 → AudioContext 또 1개 생성
   ... 계속 쌓여요 ...
10. 브라우저: "너무 많아! 더 이상 못 만들어!" 💥
```

마치 **방에 장난감을 계속 가져오는데, 절대 치우지 않는 것**과 같아요!
방이 장난감으로 가득 차면 더 이상 놀 수 없게 되죠.

**현재 코드의 문제:**
```typescript
// 😰 문제: 매번 새로운 AudioContext를 만들어요
const 소리듣기시작 = (피어, 스트림) => {
  const 오디오도구 = new AudioContext()  // 매번 새로 만듦!
  // 쓰고 나서 안 치워요...
}
```

#### 😊 어떻게 해결하나요?

**해결책: 하나의 도구를 모두가 같이 사용해요 (싱글톤 패턴)**
```typescript
// 😊 해결: 딱 하나만 만들고 다 같이 사용해요
class 오디오도구관리자 {
  static 공용도구 = null  // 모두가 같이 쓰는 도구
  
  static 도구가져오기() {
    // 아직 도구가 없으면 하나 만들기
    if (공용도구가_없거나_고장났으면) {
      this.공용도구 = new AudioContext()
    }
    // 있으면 그거 같이 쓰기
    return this.공용도구
  }
}

// 사용할 때
const 소리듣기시작 = (피어, 스트림) => {
  const 오디오도구 = 오디오도구관리자.도구가져오기()  // 공용도구 사용!
}
```

#### ✅ 해결 후 어떻게 달라지나요?

| 구분 | 해결 전 😰 | 해결 후 😊 |
|------|-----------|-----------|
| AudioContext 개수 | 사람 수만큼 계속 늘어남 (5명=5개, 10명=10개) | 항상 1개만 사용 |
| 메모리 사용량 | 시간이 지날수록 늘어남 📈 | 일정하게 유지 ➡️ |
| 1시간 후 상태 | 앱이 느려지고 버벅거림 | 처음처럼 부드러움 |
| 브라우저 제한 | "더 이상 만들 수 없음" 에러 발생 | 에러 없음 |

**비유로 설명하면:**
- **해결 전**: 색연필이 필요할 때마다 새 세트를 사서 책상 위에 쌓음 🖍️🖍️🖍️
- **해결 후**: 하나의 색연필 세트를 모두가 돌아가며 사용 🖍️ (책상이 깔끔!)

### 🔴 문제 4: 다른 앱 보다 오면 방송이 끊겨요 (모바일 백그라운드)

#### 😰 무슨 문제인가요?

핸드폰에서 다른 앱을 보면, 방송 앱이 "잠들어" 버려요.

**문제가 생기는 상황:**
```
1. 철수가 방송을 듣고 있어요 📱
2. 카카오톡 메시지가 와서 카톡을 열어요 💬
3. 방송 앱이 백그라운드로 가면서 "잠들어요" 😴
4. 인터넷 연결이 끊어져요 📵
5. 다시 방송 앱으로 돌아왔는데... 조용해요 🔇
6. "연결이 끊어졌습니다" 메시지가 떠요 ❌
```

마치 **TV 리모컨의 일시정지 버튼을 누른 것**처럼, 앱이 멈춰버리는 거예요!

#### 😊 어떻게 해결하나요?

**해결책 1: 화면을 계속 켜두기 (WakeLock)**
```typescript
// "화면 꺼지지 마세요!" 라고 핸드폰에게 부탁해요
const 화면유지하기 = async () => {
  try {
    화면잠금 = await navigator.wakeLock.request('screen')
    // 핸드폰: "알겠어, 화면 안 꺼줄게!"
  } catch (e) {
    // 일부 핸드폰은 이 기능을 지원 안 해요
  }
}
```

**해결책 2: 돌아왔을 때 바로 다시 연결하기**
```typescript
// 앱으로 돌아왔을 때 확인하고 다시 연결해요
document.addEventListener('visibilitychange', () => {
  if (document.visibilityState === 'visible') {
    // "다시 돌아왔다! 연결 확인해볼까?"
    
    if (연결이_끊어져있으면) {
      // "아, 끊어졌네. 다시 연결하자!"
      다시연결하기()
    }
  }
})
```

#### ✅ 해결 후 어떻게 달라지나요?

| 구분 | 해결 전 😰 | 해결 후 😊 |
|------|-----------|-----------|
| 카톡 확인 후 | 방송 끊김, 다시 입장해야 함 | 자동으로 다시 연결됨 |
| 잠깐 앱 전환 | 연결 끊어질 수 있음 | 화면이 꺼지지 않아 연결 유지 |
| 자동 복구 | 없음 (수동으로 다시 들어가야 함) | 있음 (1-2초 내 자동 복구) |

**비유로 설명하면:**
- **해결 전**: 방에서 잠깐 나갔다 오면 문이 잠겨있음 🚪🔒
- **해결 후**: 방에서 나갔다 와도 문이 열려있고, 잠겼어도 자동으로 열림 🚪✨

### 🟡 문제 5: WiFi에서 LTE로 바뀌면 끊겨요 (네트워크 전환)

#### 😰 무슨 문제인가요?

인터넷 연결 방법이 바뀌면 연결이 끊어질 수 있어요.

**문제가 생기는 상황:**
```
1. 집에서 WiFi로 방송을 들어요 📶
2. 외출하면서 집을 나서요 🚶
3. WiFi가 끊기고 LTE로 바뀌어요 📱
4. IP 주소가 바뀌어서 연결이 끊어져요 ❌
5. 앱이 바로 눈치채지 못해서 한참 조용해요 🔇
```

마치 **전화하다가 터널에 들어갔는데, 나와도 전화가 끊긴 상태인 것**과 같아요!

#### 😊 어떻게 해결하나요?

**해결책: 인터넷 연결 상태 지켜보기**
```typescript
// 인터넷이 다시 연결되면 바로 알아채요
window.addEventListener('online', () => {
  console.log('📶 인터넷 다시 연결됨!')
  다시연결하기()  // 바로 다시 연결 시도
})

// 인터넷이 끊어지면 사용자에게 알려줘요
window.addEventListener('offline', () => {
  console.log('📵 인터넷 끊어짐!')
  화면에_표시하기('인터넷 연결이 끊어졌어요. 잠시만 기다려주세요...')
})
```

#### ✅ 해결 후 어떻게 달라지나요?

| 구분 | 해결 전 😰 | 해결 후 😊 |
|------|-----------|-----------|
| WiFi→LTE 전환 | 5~30초 후에야 끊긴 걸 알아챔 | 즉시 알아채고 1초 내 재연결 |
| 사용자 경험 | "왜 조용하지?" 한참 기다림 | "잠깐 끊겼다 연결됨" 메시지 보임 |
| 지하철 이용 시 | 역마다 끊기고 계속 튕김 | 자동 복구로 끊김 최소화 |

**비유로 설명하면:**
- **해결 전**: 전화가 끊겨도 몰라서 "여보세요? 여보세요?" 하고 있음
- **해결 후**: 전화가 끊기자마자 바로 다시 걸어줌

### 🟡 문제 6: 후원할 때마다 느려져요 (N+1 쿼리)

#### 😰 무슨 문제인가요?

후원이 오면 "누가 보냈는지" 정보를 가져와야 해요. 그런데 비효율적으로 가져오고 있어요.

**문제가 생기는 상황:**
```
후원 1개 올 때마다:
1. "누가 보냈지?" → 데이터베이스 질문 1번
2. "받는 사람은 누구지?" → 데이터베이스 질문 1번

만약 10명이 동시에 후원하면:
→ 데이터베이스 질문 20번! 😵

100명이 동시에 후원하면:
→ 데이터베이스 질문 200번!! 💀
```

마치 **편의점에서 물건 10개 살 때, 매번 계산대에 1개씩 가져가는 것**과 같아요!

**현재 코드의 문제:**
```typescript
// 😰 문제: 매번 따로따로 물어봐요
const 후원자정보 = await 데이터베이스.질문('members', 후원자ID)  // 질문 1
const 받는사람정보 = await 데이터베이스.질문('partners', 받는사람ID)  // 질문 2
```

#### 😊 어떻게 해결하나요?

**해결책: 한 번에 다 가져오기 (JOIN)**
```typescript
// 😊 해결: JOIN으로 한 번에 다 가져와요
const 후원정보 = await 데이터베이스.질문(`
  후원내역.*,
  후원자:members(이름, 프로필사진),
  받는사람:partners(이름)
`)
// 질문 1번으로 다 가져옴!
```

#### ✅ 해결 후 어떻게 달라지나요?

| 구분 | 해결 전 😰 | 해결 후 😊 |
|------|-----------|-----------|
| 후원 1개당 | 데이터베이스 질문 2번 | 데이터베이스 질문 0번 (이미 다 있음) |
| 후원 100개 | 질문 200번 → 느림 🐢 | 질문 1번 → 빠름 🚀 |
| 후원 폭주 시 | 앱이 버벅거림 | 부드럽게 처리됨 |
| 서버 부담 | 높음 (비용 증가) | 낮음 (비용 절감) |

**비유로 설명하면:**
- **해결 전**: 마트에서 사과 10개를 1개씩 10번 계산
- **해결 후**: 마트에서 사과 10개를 카트에 담아 1번 계산

---

### 🟡 문제 7: 인터넷 연결이 너무 많아요 (채널 분산)

#### 😰 무슨 문제인가요?

실시간 기능마다 따로따로 인터넷 연결을 만들고 있어요.

**현재 상황:**
```
방 하나에 들어가면:
- 채팅용 연결 1개 🔌
- 호스트 변경용 연결 1개 🔌
- 후원용 연결 1개 🔌
- 참가자용 연결 1개 🔌
- 뮤트 상태용 연결 1개 🔌
= 총 5개의 연결!

방이 10개 있는 서버라면:
= 50개의 연결을 관리해야 함! 😵
```

마치 **TV, 에어컨, 선풍기마다 각각 다른 리모컨**을 쓰는 것과 같아요!

#### 😊 어떻게 해결하나요?

**해결책: 하나의 연결로 모든 것을 처리해요 (통합 채널)**
```typescript
// 😰 문제: 따로따로 연결
const 채팅연결 = supabase.channel('채팅-방1')
const 호스트연결 = supabase.channel('호스트-방1')
const 후원연결 = supabase.channel('후원-방1')

// 😊 해결: 하나로 통합!
const 방연결 = supabase.channel('방1')
  .on('채팅', 채팅처리)
  .on('호스트변경', 호스트처리)
  .on('후원', 후원처리)
  .on('뮤트', 뮤트처리)
  .subscribe()
// 연결 1개로 다 됨!
```

#### ✅ 해결 후 어떻게 달라지나요?

| 구분 | 해결 전 😰 | 해결 후 😊 |
|------|-----------|-----------|
| 방 1개당 연결 | 5개 | 1개 |
| 서버 부담 | 높음 (연결 많음) | 낮음 (연결 적음) |
| 관리 복잡도 | 높음 (5개 관리) | 낮음 (1개 관리) |
| 비용 | Supabase 요금 더 나감 | 요금 절감 |

**비유로 설명하면:**
- **해결 전**: TV, 에어컨, 조명마다 다른 리모컨 5개
- **해결 후**: 모든 가전을 제어하는 통합 리모컨 1개

### 🔴 문제 8: 사람이 많아지면 앱이 터져요 (Mesh 확장성)

#### 😰 무슨 문제인가요?

지금 방식은 **모든 발언자가 서로 직접 연결**해요. 사람이 많아지면 연결 수가 폭발해요!

**연결 수 계산:**
```
발언자 2명: A ↔ B = 1개 연결 ✅ 괜찮아요
발언자 3명: A ↔ B, A ↔ C, B ↔ C = 3개 연결 ✅ 괜찮아요
발언자 5명: = 10개 연결 ⚠️ 좀 많네요
발언자 10명: = 45개 연결 😰 많이 힘들어요
발언자 20명: = 190개 연결 💀 터져요!
```

마치 **반 친구 30명이 서로 실 전화기로 연결**하려는 것과 같아요!
모두가 모두와 연결하려면 전선이 너무 많아져요!

**현재 구조 (Mesh - 그물망):**
```
    A ←――――→ B
    ↕ ╲    ╱ ↕
    ↕   ╲╱   ↕
    ↕   ╱╲   ↕
    ↕ ╱    ╲ ↕
    C ←――――→ D
    
모든 사람이 서로 연결됨 (연결 6개)
```

#### 😊 어떻게 해결하나요?

**현재 대응책:**
- 발언자 수를 5~10명으로 제한 (`max_participants`)

**장기 해결책: SFU (중앙 서버) 도입**
```
SFU 구조 (중앙 서버):

    A ―――→ ┌─────┐ ←――― B
           │ SFU │
    C ←――― │서버 │ ―――→ D
           └─────┘
           
모든 사람이 서버 하나와만 연결 (연결 4개)
```

```typescript
// SFU를 쓰면...
발언자.보내기(내목소리) → SFU서버 → 모든청취자에게_전달()

// 장점:
// - 발언자 1명당 연결 1개만 필요
// - 100명, 1000명도 가능!
```

#### ✅ 해결 후 어떻게 달라지나요?

| 구분 | 해결 전 (Mesh) 😰 | 해결 후 (SFU) 😊 |
|------|-------------------|-----------------|
| 발언자 5명 | 연결 10개 | 연결 5개 |
| 발언자 20명 | 연결 190개 (거의 불가능) | 연결 20개 |
| 발언자 100명 | 연결 4,950개 (불가능!) | 연결 100개 |
| 최대 인원 | 10-15명 한계 | 수백 명 가능 |
| 핸드폰 부담 | 매우 높음 (배터리 빨리 닳음) | 낮음 |

**비유로 설명하면:**
- **해결 전 (Mesh)**: 반 친구 30명이 서로 전화선으로 연결 (전선 435개 필요!)
- **해결 후 (SFU)**: 선생님(서버)이 중간에서 말 전달 (전선 30개만 필요)
