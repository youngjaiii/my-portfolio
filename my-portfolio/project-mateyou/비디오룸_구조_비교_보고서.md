본 보고서는 현재 구현된 비디오룸 시스템과 목표 아키텍처를 비교하여, **제거해야 할 항목**과 **개발해야 할 항목**을 명확히 구분하기 위한 문서입니다.

## 2. 현재 구조 분석 (AS-IS)

### 2.1 기술 스택

| 영역 | 현재 사용 기술 | 위치/파일 |
|------|---------------|-----------|
| **송출** | WebRTC (PeerJS) | `src/contexts/VideoRoomProvider.tsx` |
| **시청** | WebRTC (PeerJS) | `src/contexts/VideoRoomProvider.tsx` |
| **시그널링** | Supabase Realtime | `src/hooks/useUnifiedStreamChannel.ts` |
| **Backend** | Supabase Edge Functions | `supabase/functions/api-stream/index.ts` |
| **채팅** | Supabase Realtime | `src/hooks/useUnifiedStreamChannel.ts` |
| **미디어 처리** | 없음 (브라우저 직접 처리) | - |
| **CDN** | 없음 | - |

### 2.2 현재 아키텍처

```
[송출자 (호스트)]
  └─ PeerJS (WebRTC)
        ↓
[PeerJS Server (peer01.mateyou.me)]
        ↓
[시청자들]
  └─ PeerJS (WebRTC) - Mesh 구조 (각 시청자가 서로 연결)
```

**특징:**
- P2P Mesh 구조로 모든 참여자가 서로 연결
- 시청자 수 증가 시 각 시청자가 모든 피어와 연결 시도
- Backend는 방 관리(생성/입장/퇴장)만 담당, 미디어 스트림은 직접 처리하지 않음

### 2.3 현재 구현 상세

#### 2.3.1 VideoRoomProvider (`src/contexts/VideoRoomProvider.tsx`)
- **역할**: PeerJS 기반 WebRTC 연결 관리
- **기능**:
  - 로컬 미디어 스트림 획득 (카메라/마이크)
  - PeerJS를 통한 P2P 연결
  - 화면 공유 지원
  - 원격 피어 스트림 수신 및 재생
- **문제점**:
  - 시청자 수 증가 시 연결 복잡도 증가 (N²)
  - 네트워크 불안정 시 재연결 로직이 복잡
  - 서버 부하 분산 불가

#### 2.3.2 Backend API (`supabase/functions/api-stream/index.ts`)
- **역할**: 방 관리 (생성/입장/퇴장/종료)
- **기능**:
  - 방 생성/조회
  - 입장/퇴장 처리
  - 하트비트 관리
  - 방 상태 관리 (scheduled/live/ended)
- **제한사항**:
  - 미디어 스트림 처리 없음
  - 스트림 키 관리 없음
  - RTMP 인증 없음

#### 2.3.3 채팅 시스템
- **기술**: Supabase Realtime
- **위치**: `src/hooks/useUnifiedStreamChannel.ts`
- **상태**: ✅ 유지 (목표 구조와 동일)

## 3. 목표 구조 (TO-BE)

### 3.1 목표 기술 스택

| 영역 | 목표 기술 | 비고 |
|------|----------|------|
| **송출 (웹/모바일)** | WebRTC (LiveKit) | 현재 PeerJS 대체 |
| **송출 (외부툴)** | RTMP | OBS/PRISM 지원 |
| **시청** | HLS | WebRTC 대체 |
| **미디어 처리** | FFmpeg | WebRTC → HLS 변환 |
| **Backend** | Node.js API | Edge Functions 대체 또는 보완 |
| **CDN** | CloudFront | HLS 스트림 배포 |

### 3.2 목표 아키텍처

```
[송출자]
 ├─ WebRTC → LiveKit Server
 └─ RTMP → Nginx RTMP Server
           ↓
        FFmpeg (WebRTC/RTMP → HLS 변환)
           ↓
          HLS 파일
           ↓
      S3 + CloudFront
           ↓
        [시청자들]
```

## 4. 비교 분석: 제거 vs 개발

### 4.1 제거해야 할 항목 (Deprecate)

#### 4.1.1 PeerJS 기반 WebRTC 연결 (시청 경로)
**위치**: `src/contexts/VideoRoomProvider.tsx`
**이유**: 
- 시청자는 HLS로 전환해야 함
- Mesh 구조는 확장성 문제가 있음

**제거 범위**:
- ✅ **제거**: 시청자용 PeerJS 연결 로직
- ✅ **제거**: 시청자용 원격 피어 스트림 수신 로직
- ✅ **제거**: Mesh 구조 관련 코드
- ⚠️ **유지**: 호스트용 WebRTC 송출 로직 (LiveKit으로 전환 전까지)

**대체 방안**:
- HLS 플레이어로 대체 (`<video>` 태그에 HLS URL 연결)

#### 4.1.2 PeerJS 서버 의존성
**위치**: `src/contexts/VideoRoomProvider.tsx` (getPeerOptions)
**이유**: 
- LiveKit으로 전환 시 PeerJS 서버 불필요

**제거 범위**:
- PeerJS 서버 설정 (`peer01.mateyou.me`)
- TURN 서버 설정 (LiveKit이 자체 제공)

#### 4.1.3 시청자용 WebRTC 스트림 처리
**위치**: 
- `src/routes/stream/video/$roomId.tsx` (시청자 스트림 표시 로직)
- `src/contexts/VideoRoomProvider.tsx` (원격 피어 스트림 수신)

**제거 범위**:
- 시청자가 호스트의 WebRTC 스트림을 수신하는 로직
- `peers` Map에서 호스트 스트림을 찾아 표시하는 로직

**대체 방안**:
- HLS URL을 받아서 `<video>` 태그에 연결
