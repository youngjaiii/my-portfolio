# Timesheet 시스템 설계 문서

## 1. 시스템 개요

### 1.1 시스템 성격

- 기존 서비스 내부에 존재하지만 논리적으로는 별도 프로그램
- 기존 User Role(`admin` | `partner` | `user`)은 접근 자격 판별용
- 본 시스템 내부에서는 추가 권한(파트너+, 파트너M)으로 동작
- 구조적으로는 MSA 형태를 가정

## 2. 권한 시스템 설계

### 2.1 기존 User Role

- `admin`
- `partner`
- `user`

> 💡 **본 시스템은 모든 사용자가 접근 가능하며, 매니저는 파트너가 아니어도 지정 가능**

### 2.2 Timesheet 전용 확장 권한

| 구분 | 성격 |
|------|------|
| 어드민 | 기존 user role admin |
| 파트너 매니저 | Timesheet 시스템 내 관리자 |
| 파트너+ | 근태 신청 주체 |
| 파트너M | 파트너 매니저 보조 관리자 |

## 3. 권한별 기능 정의

### 3.1 어드민

#### 핵심 성격

- 시스템 전권
- 관리 + 감사(Audit) 중심
- 출퇴근 자체에는 관심 없음

#### 기능

- 파트너 매니저 지정 / 해제
- 가게 생성 / 수정 / 비활성화
- 가게에 파트너 매니저 할당
- 모든 근태 / 정산 / 요청 데이터 조회
- 감사 로그 전체 조회
- 파트너 매니저 활동
- 파트너+ 활동

#### UI 진입

- 메인 → Timesheet
- 관리 + 대시보드 형태
- 감사 로그가 주요 시각 요소

### 3.2 파트너 매니저

#### 핵심 성격

- 현장 관리자
- 승인 / 수정 / 정산 조회의 중심

#### 기능

- 파트너+ 추가 / 삭제
- 파트너+ 근태 요청 수락 / 반려
- 파트너+ 근태 수정 가능
  - 수정 시 사유 입력 필수 (내부 기록용)
- 파트너+ 테이블 타임 정산 조회
- 엑셀 다운로드
- 파트너+ 출근 상태 실시간 조회

#### 로깅

모든 행동은 감사 로그 기록

- 누가
- 언제
- 무엇을
- 왜(사유)

#### UI 진입

- 메인 → Timesheet
- 관리 + 대시보드

한 화면에서 다음이 보여야 함:

- 현재 출근 중인 파트너+
- 승인 대기 요청
- 최근 활동 로그

### 3.3 파트너+

#### 핵심 성격

- 근태 요청 전용 사용자
- 조회는 최소, 행동은 단순

#### 기능

- 출근 요청
- 휴게 요청
- 휴게 종료 요청
- 퇴근 요청

요청 시 필수 선택:

- 가게
- 담당 파트너 매니저

- 자기 현재 상태 조회
- 자기 근태 이력 조회
  - 언제 출근
  - 언제 휴게
  - 언제 종료
  - 언제 퇴근

- 정산 데이터 접근 불가

#### 제한

- 동일 상태 요청 중복 불가
- 요청 처리 완료 전 추가 요청 불가

#### 로깅

- 모든 요청은 감사 로그 기록

### 3.4 파트너M

#### 핵심 성격

- 보조 관리자
- 조회 + 제한적 관리

#### 기능

- 파트너+ 리스트 조회 (간편 정보)
  - 이름만 표시
- 출퇴근 요청 수락 / 반려
- 관리 대시보드 접근 가능
- 정산 수정 불가

#### UI

- 파트너 매니저와 유사
- 일부 기능 비활성화 처리

## 4. 근태 상태 모델 (중요)

### 4.1 상태 정의

| 상태 | 설명 |
|------|------|
| `OFF` | 미출근 |
| `WORKING` | 출근 중 |
| `BREAK` | 휴게 중 |

### 4.2 상태 전이 규칙

- `OFF` → `WORKING` (출근)
- `WORKING` → `BREAK` (휴게)
- `BREAK` → `WORKING` (휴게 종료)
- `WORKING` → `OFF` (퇴근)
- `BREAK` → `OFF` (퇴근)

> ❌ **그 외 전이 불가**

## 5. UI/UX 설계

### 5.1 파트너+ 화면 (가장 중요)

#### 메인 구조

- 화면 중앙: 현재 상태 크게 표시
  - "출근 중"
  - "휴게 중"
  - "미출근"
- 하단: 상태 변경 버튼

#### 버튼 규칙

**미출근**
- 출근 버튼만 표시

**출근 중**
- 휴게
- 퇴근

**휴게 중**
- 휴게 종료
- 퇴근

#### 요청 플로우

1. 버튼 클릭
2. BottomSheet 등장
3. 가게 선택
4. 매니저 선택
5. 요청 전송
6. 요청 대기 상태 표시
7. 승인 / 반려 결과 반영

### 5.2 파트너 매니저 / 파트너M 화면

#### 메인 대시보드

- 승인 대기 요청 리스트
- 현재 출근 중 파트너+
- 오늘의 상태 요약
- 최근 수정 내역

#### 관리 페이지

- 파트너+ 관리
- 근태 기록 수정 (사유 필수)

### 5.3 어드민 화면

#### 메인 포인트

- 감사 로그
- 관리자 활동 추적
- 시스템 안정성 확인
- 출퇴근은 "데이터 중 하나"일 뿐
- 관리와 통제가 중심

## 6. 감사 로그 설계

### 공통 로깅 대상

- 출근 / 휴게 / 퇴근 요청
- 승인 / 반려
- 근태 수정
- 파트너+ 추가 / 삭제
- 파트너 매니저 지정

### 로그 구조

- `actor` - 행위자
- `role` - 역할
- `action` - 행동
- `target` - 대상
- `timestamp` - 시간
- `reason` - 사유 (선택)

## 7. 개발 플랜 (단계별)

### 1단계

- 권한 모델 설계

### 2단계

- 파트너 매니저 대시보드
- 실시간 상태 조회

### 3단계

- 감사 로그 시스템
- 어드민 대시보드
- 엑셀 다운로드

### 4단계

- 근태 수정 + 사유 기록
- 권한 세분화
- 운영 안정화

## 핵심 요약

이 시스템의 본질은 **"출근을 기록하는 앱이 아니라, 사람의 행동을 통제하고 추적하는 관리 시스템"**이다.

지금 구조는:

- 권한 분리 명확하고
- 상태 모델 단순하며
- UI가 역할에 맞게 잘 갈라진다

다음으로 가면 가장 좋은 건 **ERD + 상태 전이 다이어그램 + API 명세서**다.  
이 정도 기획이면 개발자 입장에선 거의 설계도 완성 단계다.
